Final Project Track B: Networked Sensor-Controlled Game

DRAFT — for student review only. Requirements, deadlines, and rubric details may change before finalization. The overall scope and deliverables are stable enough for team formation and early planning.

At a Glance

In one sentence. Build a multiplayer card or dice game (Blackjack, Yahtzee, or similar) where players control the action by shaking, tilting, or flicking their phones, and moves are exchanged between computers through a cloud channel so teammates can play across machines (even on different networks).

What stays from Mini-Projects 2 through 5. Resultant acceleration (MP2), threshold detection (MP3), the SensorDataAnalyzer class (MP4), App Designer and separation of concerns (MP5).

What is new. Live gesture detection on a moving window of sensor data, game state management, and automatic turn passing between computers through ThingSpeak.

Core deliverable. A working game played across two machines, with gestures replacing mouse clicks; a short Design Memo (2 to 3 pages) explaining your game rules, architecture, and tradeoffs; and a 6-minute YouTube video following the format in the Assessment Guide.

Project Description

You will design a multiplayer card or dice game in which players' actions are controlled by sensors on their smartphones and moves are exchanged between computers through a shared cloud channel on ThingSpeak. You will build the game's GUI using App Designer and its sensor input using the class-based approach from Mini-Projects 4 and 5. The game must be playable across two or more computers on the same network or different networks; players do not need to be in the same room.

The project builds directly on your Mini-Projects. Your SensorDataAnalyzer class from MP4 is the starting point for sensor acquisition. Your gesture detection reuses the threshold-detection pattern from MP3 and the resultant-acceleration formula from MP2. Your GUI extends App Designer work from MP5 and is held together by the separation-of-concerns discipline that MP5 introduced.

In addition to App Designer skills from MP5, this project will expose you to the following additional topics:

  • Detecting meaningful events from live sensor data using threshold-based gesture recognition.
  • Using a cloud message-passing platform (ThingSpeak) to coordinate state between independent machines.
  • Designing an application with multiple clean concerns (sensor input, game rules, network communication, GUI) and keeping them decoupled.
  • Handling automatic turn passing without direct connections between players' computers.

Why This Project

The mini-projects in this course taught you a specific engineering discipline: keep sensor acquisition, analysis, and user interface separate. Mini-Project 5 made the case in a single-user setting; the rule felt slightly abstract because a single-user app could plausibly have been written without it. In Track B the discipline becomes necessary. A networked game has four concerns: what the sensor is telling you, what the game rules say, what the other player's computer is telling you, and what the user sees on screen. Tangling any two of these together creates bugs that are very hard to fix. This project rewards students who took the MP5 lesson seriously.

Project Requirements

Your project is divided into required core functionality, a required reach feature, and a team-chosen reach feature.

Your project should implement a card or dice game of your choosing. The game must be non-trivial and have some strategic component. Examples of acceptable complexity: Blackjack, Yahtzee (or a simplified version), or a scaled-down Boggle. The game War is too simple because there are no strategic choices; every turn is forced by the cards.

Core Functionality (Must-Have)

The following functionality is required:

  1. A reasonably attractive user interface built in App Designer. The UI shows the game state clearly (cards, dice, scores, turn indicator, whose turn it is) and updates automatically as the state changes.

  2. Sensor-driven control. All in-game actions that a player takes must be controlled by smartphone sensors. Keystrokes and mouse input are allowed only to connect to ThingSpeak, to start the game, and to end the game. Examples of acceptable sensor controls:

  3. Shake the phone to roll dice or deal cards.
  4. Tilt the phone forward to hit, backward to stand (Blackjack).

  5. Flick the phone in a direction to select a choice.

  6. Gesture detection must use threshold detection on resultant acceleration. Compute resultant acceleration using the formula from Mini-Project 2. Apply a threshold function modeled on Mini-Project 3's detectRapidAcceleration. Smoothing (MP5's movmean) is encouraged to reduce false triggers.

  7. The gesture detector must live in its own class. Follow the Mini-Project 4 pattern. You may extend SensorDataAnalyzer with a detectGesture method, or compose a new class on top of it. The choice is yours; defend it in your design memo.

  8. Automatic turn passing. When a player completes their turn, play passes to the next player without anyone clicking anything. In a networked session (see required reach) this means the other player's app notices the state change on ThingSpeak and updates its own UI automatically.

  9. Separation of concerns. Your project must keep these four concerns in separate files or classes: sensor acquisition, gesture detection, game rules, and network communication. A GUI callback may orchestrate them but must not implement them. This is explicitly graded.

Required Reach Feature: Remote Gameplay via ThingSpeak

The game must be playable by two or more players, each on a different computer. State changes pass through a shared ThingSpeak channel, not direct machine-to-machine connections. Players may sit next to each other in a classroom or be on different Wi-Fi networks. Turn passing remains automatic in remote mode: the other player's app polls the channel and updates when a new move appears.

See Appendix A for the recommended message-bus pattern.

Chosen Reach Feature: Pick One

You select one additional reach feature in addition to core and the remote gameplay reach. We will consider the creativity and complexity of your reach goal when evaluating your project. Some suggestions:

  • Sound effects and animations. Provide sound and visual animation for at least three in-game events (for example, deal, roll, win). Animations must be visible; silent event logs do not count.
  • Multi-sensor input selection. Different players may independently select which sensor channel controls their play (for example, player 1 uses shake, player 2 uses tilt). The game logic does not change. This is where the MP5 separation-of-concerns lesson becomes visible: if your gesture detector is properly decoupled from the game rules, this reach feature is a two-line change; if not, it is a rewrite.
  • Non-trivial randomness. Incorporate randomness that materially affects play beyond a standard shuffle (for example, random event cards, dynamic rule modifiers, chance-based scoring multipliers). Document the mechanism in your design memo.
  • Spectator view. A third computer can connect and watch the game in progress without being able to play. Useful for demonstrations.
  • Persistent leaderboard. Game results are saved across sessions (either to a local .mat file or to a second ThingSpeak channel). The title screen shows a top-five leaderboard that updates automatically after each game.
  • Personal gesture calibration. At the start of a session, each player performs a short calibration routine (shake gently, shake hard) from which the app computes per-player thresholds. Document whether this reduces false triggers compared to a fixed threshold.
  • Chat channel. Players can send short preset messages (for example, "Your move", "Nice", "Good game") via a secondary ThingSpeak channel. Messages appear in both apps.

Notes on Scope

  • Play the game on paper first. Before writing any code, write the rules in full and play a round or two with real cards or dice. Games that look obvious are full of edge cases; discover them on paper, not in MATLAB.
  • Test turn passing before testing any game logic. Your first working version should be a trivial two-app demo where one phone's shake increments a counter on both apps via ThingSpeak. Once that works, wrap the real game around it.
  • ThingSpeak rate limits apply even on the paid subscription. Plan around one update per second per channel. Your game must tolerate a couple of seconds of turn latency. Do not design reaction-time or real-time-competitive games.
  • Keep the gesture thresholds tunable. During testing you will discover that your initial thresholds fire too much or too little. Build a slider or an edit field into a hidden developer panel so you can adjust without editing code.
  • Two team members must test on two machines during development. Do not leave "try it on two computers" for the demo.

Academic Integrity

All code must be written by your team. You may use AI assistants as learning aids consistent with prior mini-projects. Final submitted code, the memo, and the video must reflect your own work. External libraries or reference sources must be cited in the memo's appendix.

Appendix A: ThingSpeak as a Message Bus

ThingSpeak is a publish-and-read platform, not a direct push one. The basic pattern for a two-player game:

  1. You create a single game-state channel (or a short-lived channel per game session).
  2. Each player's app holds a "last read entry ID" locally.
  3. When a player makes a move, the app writes the new move to the channel.
  4. Each app polls the channel on a timer (for example, once per second). On each poll, it asks ThingSpeak for entries newer than its last-read ID. If there are any, it updates its local game state from them and updates its last-read ID.
  5. Polling runs continuously; the game does not require a central "server" process.

Because both apps are writing to and reading from the same channel, a shared convention about how to encode a move is important. A simple approach is a single field that carries a compact move string (for example, "P1:hit", "P1:stand", "P2:deal"). More sophisticated games use multiple fields (player ID, action, action argument) per write.

For reading newer entries, consult the MathWorks documentation for thingSpeakRead; the entries and outputFormat arguments are the ones you want.

Appendix B: Recommended Architecture

A working, clean four-concern separation looks approximately like this. The specific class names are up to you.

SensorDataAnalyzer.m (from Mini-Project 4; extend it or leave it as-is).
Acquires and exposes sensor data.

GestureDetector.m (new).
Holds a SensorDataAnalyzer instance. Has a detect method that returns the gesture, if any, observed in the most recent sample window. Encodes your threshold logic.

GameState.m (new).
Holds the full state of the game: whose turn it is, current hands or dice, scores, history. Exposes methods like applyMove, isGameOver, legalMoves. Has no knowledge of sensors or networking.

NetworkMessenger.m (new).
Wraps your ThingSpeak read and write logic. Exposes methods like sendMove and checkForRemoteMoves. Has no knowledge of the game rules.

GameApp.mlapp (new).
App Designer GUI. Owns instances of GestureDetector, GameState, and NetworkMessenger. Its callbacks are short: on a timer tick, ask GestureDetector if a gesture occurred, check NetworkMessenger for remote moves, apply any new moves to GameState, and redraw the UI from the new GameState.

If you keep this separation, the "multi-sensor input" reach feature becomes trivial (swap out GestureDetector's behavior without touching anything else). If you skip the separation, that reach feature becomes impossible without a rewrite.

Appendix C: Reusing Mini-Project 3 for Gesture Detection

Your GestureDetector class should reuse the pattern from Mini-Project 3 directly. Shake detection is functionally identical to MP3's rapid acceleration detection. The differences are:

  • In MP3 you processed a completed data array. In Track B you process a moving window of the most recent samples.
  • In MP3 the output was a list of timestamps above threshold. In Track B the output is a single boolean: did a gesture happen in the last window.
  • Smoothing (Mini-Project 5's movmean) matters more here. A raw accelerometer reading can briefly spike above your threshold during normal handling; smoothing filters these out so your game is not constantly firing phantom shakes.

A minimum viable detector:

function fired = detect(obj, window)
    % window is the latest N accelerometer samples (Nx3)
    Ax = window(:, 1); Ay = window(:, 2); Az = window(:, 3);
    A = sqrt(Ax.^2 + Ay.^2 + Az.^2);     % MP2 resultant
    A = movmean(A, 5);                    % MP5 smoothing
    fired = any(A > obj.Threshold);       % MP3 thresholding
end

Tuning obj.Threshold is the tradeoff you analyzed in Mini-Project 3: too low fires on nothing, too high misses real shakes. Document your chosen value in your design memo.

Appendix D: MATLAB Timer Pattern

Your app needs two concurrent activities: reading sensor data and polling ThingSpeak. Both are driven from a timer object in your App Designer app.

% In startupFcn or a "Connect" button callback:
app.MainTimer = timer( ...
    'ExecutionMode', 'fixedRate', ...
    'Period', 1.0, ...
    'TimerFcn', @(~,~) app.mainLoop());
start(app.MainTimer);

The mainLoop method does the following, in order:

  1. Asks GestureDetector for a gesture in the latest sensor window.
  2. If a gesture is detected and it is this player's turn, apply the corresponding move to GameState, send the move via NetworkMessenger, and redraw the UI.
  3. Asks NetworkMessenger for new remote moves. If any arrived, apply them to GameState and redraw the UI.

If mainLoop takes longer than the timer Period, calls will queue up and the app will feel laggy. Keep the body short.
