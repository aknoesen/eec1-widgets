1. The Two Integration Functions

Function Returns Use when
cumtrapz(t, y) Array — running total at every time step You need the full trajectory (velocity vs. time, displacement vs. time)
trapz(t, y) Scalar — total area under the curve You need one number (e.g., total distance)

The last element of cumtrapz equals trapz — both give the total, but only cumtrapz gives every point along the way.

In MP4 you need cumtrapz — the displacement plot requires the trajectory, not just the final value.

2. Double Integration: Acceleration → Displacement

Call integrate() twice on the same SensorDataAnalyzer object:

analyzer.integrate();   % call 1: acceleration  →  velocity   (m/s²  →  m/s)
analyzer.integrate();   % call 2: velocity       →  displacement  (m/s  →  m)

Each call overwrites TargetSignal with the integrated result. The timestamps stay unchanged. Inside integrate():

obj.TargetSignal = cumtrapz(obj.TargetSignalTs, obj.TargetSignal);

Initial condition: cumtrapz starts the integral at zero. This assumes the phone is at rest and at position zero at t = 0. If either condition is violated, the error persists through both integrations.

3. Why Drift Grows — The Bias Math

Even a small constant bias b in the acceleration (e.g., gravity leaking into the X-axis, a tilted phone) causes displacement to drift away from the truth:

Integration Error introduced
accel → velocity b · t   (linear in time)
velocity → displacement ½ · b · t²   (quadratic in time)

Consequence: a 5% bias in acceleration produces a 5% error in displacement — but that 5% is of a quantity that grows with t². When time doubles, the absolute error quadruples. This is why dead reckoning fails over long time windows.

4. Removing the Bias Before Integrating

The fix: subtract the sensor’s resting baseline before the first cumtrapz call. During the stationary window at the start of the recording, true net acceleration ≈ 0, so the mean captures only the bias (gravity + DC offset):

g_offset = mean(Az_stationary);   % measured during the still window at start
az_net   = Az - g_offset;         % remove gravity/DC — net motion only

This only works if the phone is genuinely still during the reference window. Any residual offset after subtraction still grows quadratically in displacement.

5. What You Write in MP4

In SensorDataAnalyzer.integrate():

function integrate(obj)
    if isempty(obj.TargetSignal)
        error('No signal loaded. Call setTargetSignal first.');
    end
    obj.TargetSignal = cumtrapz(obj.TargetSignalTs, obj.TargetSignal);
end

In your experiment script:

analyzer = SensorDataAnalyzer();
analyzer.logSensorData(15, 3);          % 15 s recording, 3 s delay
analyzer.setTargetSignal("X-Accel");    % pick the forward axis
analyzer.plotTargetSignal();            % raw acceleration — save as rawAcceleration.png

analyzer.integrate();                   % accel → velocity
analyzer.plotTargetSignal();            % save as velocityEstimate.png

analyzer.integrate();                   % velocity → displacement
analyzer.plotTargetSignal();            % save as displacementEstimate.png

What to look for in the displacement plot: if the final value is far from zero after an out-and-back walk, that is drift caused by accumulated bias. The further from zero, the larger the effective bias in your signal.
