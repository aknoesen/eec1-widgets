Pre-Lab Part 1: AI Exploration

This lab is built around a load-bearing concept: the sensitivity/specificity tradeoff in threshold detection. You need to understand this tradeoff before the lab session, not because it will be tested immediately, but because you will use it to make a principled choice about your threshold value during Lab Steps 2 and 3. Students who arrive without this understanding will set thresholds by trial and error. Students who arrive with it will know what they are trading off each time they adjust a number.

Step 1: Open an AI assistant of your choice.

Step 2: Use the following as your opening prompt:

"I am a first-year engineering student building a collision detection algorithm that flags moments when a sensor reading exceeds a threshold value. I get to choose the threshold. What I don't understand is: how do I choose a good threshold? If I make it too low, what goes wrong? If I make it too high, what goes wrong? Is there a principled way to decide, or is it always trial and error? I've heard the terms sensitivity and specificity. Can you explain what those mean in the context of a threshold detector, without using statistics notation?"

Step 3: Ask the AI the following question:

"Can you give me a real-world example (like an airbag system) where the consequences of a false positive and a false negative are different? Which error is worse in that case?"

In your notes, record the example the AI gave you and your own assessment of which error is worse in that application. This becomes Problem 1(a) of your pre-lab submission. At least 60 words.

Step 4: Follow up with this question to extend your understanding:

"Is there a way to find a threshold that minimizes both types of errors simultaneously, or is that mathematically impossible? Why?"

Then verify that the AI confirms sensitivity and specificity are in tension, meaning you cannot simultaneously maximize both. If the AI's answer is ambiguous, push back and ask it to clarify.

Step 5: Answer the following in your own words. These become Problems 1(b)–1(e) of your pre-lab submission. You will also need them when choosing thresholds during the lab session:

  • 1(b): What happens if you set the threshold too low? (At least 30 words.)
  • 1(c): What happens if you set the threshold too high? (At least 30 words.)
  • 1(d): What is the name for this tradeoff?
  • 1(e): What determines which type of error is more acceptable in a given application? (At least 30 words.)

Step 6: Write a 3–5 sentence summary of what you learned. This becomes Problem 1(f) of your pre-lab submission. Your summary must connect the concepts above into a coherent explanation, not just list them.

Why this matters now: In Lab Step 3, you will choose a threshold for each of three experiments and record your reasoning before seeing the results. After all three runs, you will observe whether the same threshold worked across all activities. That observation is the empirical basis for your formal reflection. If you do not understand the tradeoff before the lab, the three-experiment structure will produce data without meaning.

Lab Step 1: Data Collection

Background

This lab uses the same data format as Lab 2: a four-column array called sensorData with X, Y, Z acceleration in columns 1–3 and timestamps in column 4. You will collect this data using the same mobiledev / accellog approach from Lab 2.

One important difference: the physical activity you perform during collection matters here in a way it did not in Lab 2. You need clear rapid acceleration events in your data: moments of sharp acceleration against a background of slower movement. If your data is too uniform, the detection algorithm will have nothing meaningful to flag regardless of what threshold you choose.

Recommended activities (choose one per run, use different ones across your three runs):
- Walking at a normal pace, then suddenly stopping
- Slow arm movement followed by a sharp shake or flick
- Walking pace followed by a jump
- Slow rotation of the phone followed by a rapid change in direction

The key feature in each case: there must be a contrast between baseline motion and a sharp event.

Lab Step 2: Write the Detection Function

Create detectRapidAcceleration.m in your Lab3 folder.

Function Signature

function [ts, A, avgA] = detectRapidAcceleration(sensorData, threshold)

Inputs

Lab Step 2: Write the Detection Function
Input Description
sensorData Nx4 array; columns 1–3 are X/Y/Z acceleration (m/s²), column 4 is timestamps
threshold Scalar detection threshold (m/s²); events above this value are flagged

Outputs

Lab Step 2: Write the Detection Function
Output Description
ts Timestamps where resultant acceleration exceeds threshold
A Resultant acceleration values at those timestamps
avgA Mean of A; set to 0.0 if no events were detected (i.e., A is empty)

What to Implement Inside the Function

  1. Extract the three acceleration columns and the timestamp column from sensorData
  2. Compute resultant acceleration using the same formula as Lab 2:
    matlab A_resultant = sqrt(Ax.^2 + Ay.^2 + Az.^2);
  3. Find all indices where resultant acceleration exceeds threshold:
    matlab idx = A_resultant > threshold;
  4. Extract timestamps and resultant values at those indices:
    matlab ts = timestamps(idx); A = A_resultant(idx);
  5. Compute avgA. Use isempty to check whether any events were detected, and mean to average the values if they exist:
    matlab if isempty(A) avgA = 0.0; else avgA = mean(A); end

Important: The function must handle the case where no events are detected: ts and A will be empty arrays, and avgA must be 0.0. Do not leave this unhandled; it will cause errors in the analysis script.

Lab Step 3: Write the Analysis Script

Create analyzeData.m in your Lab3 folder.

This script loads your sensor data, calls your detection function, and produces a visual report. You will run it three times, once for each activity, producing three separate, saved plots.

What the Script Must Do (for each run)

  1. Load sensorData.mat:
    matlab load('sensorData.mat')

  2. Convert timestamps to elapsed time starting from zero

  3. Call your detection function with a chosen threshold:
    matlab [ts, A, avgA] = detectRapidAcceleration(sensorData, threshold);

  4. Plot resultant acceleration vs. elapsed time

  5. Add a vertical red line at each flagged timestamp using xline; the documentation shows how to control line style, color, and labels:
    matlab xline(ts, 'r', 'LineWidth', 1.5);

  6. Add a horizontal dashed line at the threshold value using yline; the documentation shows how to add an automatic label:
    matlab yline(threshold, '--k', 'Threshold');

  7. Add a text annotation showing avgA. Place it in a visible but non-overlapping location, for example the upper-left corner of the axes. Use the text function; the documentation's Units property options are useful for normalized coordinate positioning.

  8. Label axes with units (elapsed time in seconds, acceleration in m/s²)

  9. Include the activity name and chosen threshold in the subplot title

  10. Add your name to the figure title using sgtitle:
    matlab sgtitle('Rapid Acceleration Analysis: Your Name Here')

  11. Save the figure as a PNG using exportgraphics:
    matlab exportgraphics(gcf, 'rapidAcceleration_run1.png', 'Resolution', 150)
    Use run1, run2, run3 for each experiment.
