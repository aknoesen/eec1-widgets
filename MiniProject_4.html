Background: Object-Oriented Programming

Before you open the skeleton file, read this — it will make the code structure legible.

A class is a template that bundles data (called properties) and operations (called methods) together into a single named object. When you create an instance of a class, you get an object that carries both its own data and the functions that operate on that data.

Properties can be public (accessible and modifiable from outside the class) or private (Access property attributes — controlling visibility of class properties) (visible only inside the class). The mobiledevConnection property in SensorDataAnalyzer is private — this means you cannot accidentally overwrite or break the sensor connection from your script. The internal data (TargetSignal, TargetSignalTs) is public so your script can read results after calling a method.

Why wrap mobiledev in a class at all? The class adds a consistent interface: every step in the pipeline has a named method with defined inputs and outputs. It also adds error checking — setTargetSignal throws a meaningful error if you call it before logging, rather than silently producing garbage. You are not expected to design OOP systems in this course; you are expected to read the structure, implement the methods in the skeleton, and understand why the structure is organized the way it is.

Constructor: SensorDataAnalyzer

The constructor runs when you call obj = SensorDataAnalyzer(). It must:

  • Create a mobiledev object and store it in the private property mobiledevConnection

Because mobiledevConnection is private, code outside the class cannot overwrite or accidentally reassign the sensor connection. The connection is created once and lives for the lifetime of the object.

logSensorData(obj, timeSeconds, delaySeconds)

  • Waits delaySeconds before starting — use pause(delaySeconds) (pause docs — suspend execution for a specified number of seconds) — to give you time to get the phone oriented and steady
  • Starts logging on mobiledevConnection
  • Waits timeSeconds, then stops logging
  • Retrieves accelerometer data using accellog(mobiledevConnection)
  • Stores the raw data array internally (in a private property) for use by setTargetSignal

setTargetSignal(obj, signalName)

Accepts one of four string arguments: "X-Accel", "Y-Accel", "Z-Accel", "R-Accel" (resultant).

  • Extracts the corresponding column from the logged data array:
  • "X-Accel" → column 1
  • "Y-Accel" → column 2
  • "Z-Accel" → column 3
  • "R-Accel" → sqrt(Ax.^2 + Ay.^2 + Az.^2)
  • Stores the signal in the public property TargetSignal
  • Stores elapsed timestamps (zeroed from first sample) in the public property TargetSignalTs
  • Throws an error using error() if:

MathWorks Docs: error — throw an error and display a formatted message
- signalName is not one of the four valid strings
- No logging session has been run yet (raw data is empty)

plotTargetSignal(obj)

  • Plots TargetSignal vs. TargetSignalTs
  • Labels axes: time axis in seconds, signal axis in m/s² (or m/s or m depending on integration state — label what the signal actually is at the time of plotting)
  • Adds a descriptive title
  • Throws an error if TargetSignal is empty

getTargetSignal(obj)

  • Returns [TargetSignal, TargetSignalTs]
  • Throws an error if TargetSignal is empty

Lab Step 2: Add Numerical Integration

Background: Dead Reckoning

Dead reckoning is a navigation technique: estimate your current position from a known starting point by accumulating measured changes over time. Ships used it for centuries before GPS; inertial navigation systems in aircraft and spacecraft still use it today.

The chain works as follows: velocity is the integral of acceleration over time; position is the integral of velocity over time. If you measure acceleration continuously and integrate it twice from a known starting state, you get an estimate of displacement.

In this lab, the starting conditions are: phone at rest (velocity = 0) at t = 0, starting position = 0. Any deviation from these conditions at the start — or any error in the acceleration measurement — enters the estimate and stays there. That is the core problem with dead reckoning: errors do not cancel, they accumulate. You will see this directly in your experiment.

integrate(obj) method

Add this method to your SensorDataAnalyzer class:

  • Numerically integrates TargetSignal with respect to TargetSignalTs using cumtrapz — consult the documentation for the two-argument form, which is required here to handle non-uniform timestamp spacing correctly:
    matlab cumtrapz(obj.TargetSignalTs, obj.TargetSignal)
    This handles nonuniform timestamp spacing correctly — important because smartphone sensors do not sample at perfectly uniform intervals.
  • Overwrites TargetSignal with the integrated result
  • Throws an error if TargetSignal is empty

Calling integrate once transforms acceleration → velocity. Calling it a second time transforms velocity → displacement. The timestamps remain unchanged between calls.

Lab Step 3: Dead Reckoning Experiment

Hold your phone with the positive X-axis pointing forward along your direction of travel. You identified which physical direction is positive X in Lab 1 — check your notes before proceeding.

Procedure

Run the following steps in your MATLAB script:

  1. Create a SensorDataAnalyzer object:
    matlab analyzer = SensorDataAnalyzer();

  2. Log 15 seconds of data with a 3-second delay:
    matlab analyzer.logSensorData(15, 3);
    Use the 3-second delay to settle the phone into position before data collection starts.

  3. During the 15-second logging window, follow this sequence:

  4. Stand still for approximately 2 seconds
  5. Walk forward approximately 5 paces at a normal walking pace
  6. Pause briefly at the far point
  7. Walk back to your starting position

  8. Stand still for the remaining time

  9. Set the target signal to X-axis acceleration:
    matlab analyzer.setTargetSignal("X-Accel");

  10. Plot the raw X-axis acceleration. Before saving, add your name to the title. Save as:
    rawAcceleration.png

  11. Call integrate once to produce estimated velocity:
    matlab analyzer.integrate();
    Plot the result. Add your name to the title. Save as:
    velocityEstimate.png

  12. Call integrate a second time to produce estimated displacement:
    matlab analyzer.integrate();
    Plot the result. Add your name to the title. Save as:
    displacementEstimate.png
