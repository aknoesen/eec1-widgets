Background: AppDesigner and Event-Driven Programming

Concept box: read before starting Step 1

AppDesigner is MATLAB's tool for building GUIs without writing layout code by hand. You drag and drop components onto a canvas, arrange them visually, and then write the callback functions that respond to user actions.

A callback is a function that runs in response to a user action (a button press, a dropdown change), not in sequence from top to bottom like a script, but triggered by events. This is called event-driven programming: the order in which code executes is determined by what the user does, not by the structure of the file.

The analysis logic inside SensorDataAnalyzer runs sequentially within its methods; only the triggering of those methods is event-driven. A callback that calls app.Analyzer.integrate() is brief: it triggers a method, re-plots the result, and exits. All the integration math lives in the class, untouched by the GUI layer.

This is exactly why the separation matters. The analysis methods work correctly regardless of whether they are called from a GUI callback, the command line, or a test script. You already verified this in Lab 4: you tested integrate() from the command line before any GUI existed. That was only possible because the logic was in the class.

Lab Step 2: Add Startup Function and App Property (Code View)

Switch to Code View by clicking the Code View button at the top of AppDesigner.

Add a public property called Analyzer. In the Code View editor, locate the properties block or add one:

properties (Access = public)
    Analyzer  % SensorDataAnalyzer instance
end

Add a startup function. In the Component Browser or Callbacks menu, add a startupFcn callback:

MathWorks Docs: startupFcn: App Designer startup function; see "Manage Code in App Designer Code View" for how and when to use it

function startupFcn(app)
    app.Analyzer = SensorDataAnalyzer();
end

This creates a SensorDataAnalyzer instance when the app launches and stores it as app.Analyzer. Every callback you write will use app.Analyzer to call analysis methods.

Run the app using the green play button. If it launches without errors, the constructor is working correctly and MATLAB can find SensorDataAnalyzer.m. If you see a "class not found" error, confirm both files are in the same folder and that folder is the MATLAB working directory.

Lab Step 3: Add Callbacks (build and test incrementally)

Rule for this step: Add one callback, run the app, test it, then move to the next. Do not write all callbacks before testing any of them. If an error appears, fix it before moving on; debugging one callback at a time is far easier than debugging five at once.

Step 3.1: Collect Data Button

In Design View, right-click the Collect Data button and select Callbacks → Add ButtonPushedFcn Callback (ButtonPushedFcn docs). AppDesigner switches to Code View and places the cursor inside the new callback function.

The callback should:

  1. Read the Duration value from the edit field: app.DurationEditField.Value
  2. Read the Delay value from the edit field: app.DelayEditField.Value
  3. Call app.Analyzer.logSensorData(duration, delay)
  4. Read the dropdown value: app.TargetSignalDropDown.Value
  5. Call app.Analyzer.setTargetSignal(dropdownValue)
  6. Plot the signal on the UIAxes:
    matlab plot(app.UIAxes, app.Analyzer.TargetSignalTs, app.Analyzer.TargetSignal)

Note on plotting: The plotTargetSignal method in Lab 4 plots to the current figure. For the GUI, you must plot directly to app.UIAxes rather than calling plotTargetSignal. Use the plot call above, or modify plotTargetSignal to accept a UIAxes handle as an argument.

Run the app. Press Collect Data with a short duration (5 seconds). Wave your phone. The signal should appear in the axes. If the axes remain blank, check that setTargetSignal is being called with a valid signal name.

Step 3.2: Set Target Signal Dropdown

Add a ValueChangedFcn callback to the dropdown. (ValueChangedFcn docs) When the user changes the dropdown:

  1. Call app.Analyzer.setTargetSignal(app.TargetSignalDropDown.Value)
  2. Re-plot on UIAxes: plot(app.UIAxes, app.Analyzer.TargetSignalTs, app.Analyzer.TargetSignal)

Run and test: collect data, then change the dropdown; the axes should update to the newly selected signal.

Step 3.3: Smooth Signal Button

Add a ButtonPushedFcn callback to the Smooth Signal button.

The callback should:

  1. Read the Window Size value: app.WindowSizeEditField.Value
  2. Call app.Analyzer.smooth(windowSize)
  3. Re-plot on UIAxes

If smooth is not yet implemented in SensorDataAnalyzer: Add the following method to SensorDataAnalyzer.m and save the file before running the app:
```matlab
function smooth(obj, windowSize)
obj.TargetSignal = movmean(obj.TargetSignal, windowSize);

MathWorks Docs: movmean: moving average using a sliding window; see documentation for endpoint handling options
end
`` SaveSensorDataAnalyzer.m`, then return to the app and test the Smooth Signal button.

Run the app. Collect data. Press Smooth Signal. The signal should visibly smooth; try a Window Size of 10 to confirm the effect is visible.

Step 3.4: Integrate Target Signal Button

Add a ButtonPushedFcn callback to the Integrate Target Signal button.

The callback should:

  1. Call app.Analyzer.integrate()
  2. Re-plot on UIAxes: plot(app.UIAxes, app.Analyzer.TargetSignalTs, app.Analyzer.TargetSignal)

Run and test. After integrating acceleration, the signal should show velocity (or displacement after a second integration). The shape and units will differ from the raw signal; this is expected.

Step 3.5: Reset Target Signal Button

The Reset button restores the original unprocessed signal after smoothing or integration. To enable this, SensorDataAnalyzer must store a copy of the original signal.

Modify SensorDataAnalyzer.m:

  1. Add a private property OriginalTargetSignal:
    matlab properties (Access = private) OriginalTargetSignal end

  2. In the setTargetSignal method, store a copy immediately after extracting the signal:
    matlab obj.OriginalTargetSignal = obj.TargetSignal;

  3. Add a reset method:
    matlab function reset(obj) obj.TargetSignal = obj.OriginalTargetSignal; end

Save SensorDataAnalyzer.m. Add a ButtonPushedFcn callback to the Reset Target Signal button:

  1. Call app.Analyzer.reset()
  2. Re-plot on UIAxes

Run and test. Collect data, smooth it, then press Reset; the signal should return to the original unprocessed state, visually matching what was shown immediately after collection.
